# PostAI 海报生成优化日志

日期：2026-06-03

## 问题判断

这次阅读了前端入口、后端编排链路、ContentExtractor、StyleDirector、SpatialLayoutPlanner、HTMLPainter、VLMCritic 和测试用例。前端只负责提交 prompt、展示图片和日志，没有对海报做会导致单调的二次裁剪；核心问题在后端生成链路。

主要问题如下：

- LLM 生成温度固定为 `0.2`，对 HTML/CSS 海报这种创意任务过于保守，容易产出安全但无趣的布局。
- LayoutPlanner 虽然强调“poster”，但没有明确要求视觉层次、材质、图形系统和构图复杂度，模型容易退化成渐变背景、居中文字、小图标。
- 本地 fallback HTML 模板过于简单，典型输出是大面积渐变、一个淡色圆形 SVG、底部两行文字，像占位图而不是海报。
- 无 VLM 时的 heuristic critic 只检查文字和 HTML 结构，无法识别“模板化、稀薄、单调”的视觉问题，因此容易提前通过。
- ContentExtractor fallback 对纯文字/极简需求仍可能硬塞主视觉，导致不同海报类型被推向相似结构。

## 已实现优化

### 1. LLM 生成参数分层

修改文件：

- `backend/app/core/config.py`
- `backend/app/core/llm_client.py`

新增配置：

- `LLM_TEMPERATURE`，默认 `0.35`，用于结构化 JSON 输出。
- `LLM_RAW_TEMPERATURE`，默认 `0.78`，用于 HTML/CSS 原始生成。

效果：

- Content/Style 等结构化 agent 仍保持稳定。
- HTML 海报生成获得更高创意自由度，减少低温导致的保守模板感。

### 2. 强化内容与艺术方向提示词

修改文件：

- `backend/app/agents/content_extractor.py`
- `backend/app/agents/style_director.py`

优化点：

- ContentExtractor 要求输出可生成视觉的 `visual_subjects`，例如裁切物体、符号、纹理场、图案系统、框架、抽象几何等。
- StyleDirector 要求给出可执行的海报视觉想法，例如巨大裁切字体、分割场、纸张纹理、模块网格、拼贴、符号几何、动势斜线结构。
- 明确避免默认退化成“普通渐变背景 + 居中文字”。

### 3. 强化 LayoutPlanner 海报丰富度约束

修改文件：

- `backend/app/agents/layout_planner.py`

新增 `POSTER RICHNESS BUDGET`：

- 非极简需求下至少包含 5 个可见视觉层：底色场、纹理/噪声/网格、巨大裁切文字或符号、方向性形状、焦点视觉、metadata strip、框线/规则线、前景文字层级等。
- 禁止把单个居中图标、小 SVG 圆圈或单一渐变当成完整视觉概念。
- 要求标题具备明确排版处理，并避免编造日期、地点、价格、赞助商等事实信息。

### 4. 重写 fallback HTML 模板

修改文件：

- `backend/app/render/html_painter.py`

重写了 4 类 fallback 模板，同时保留测试依赖的结构标记：

- `type_only`：纯字排版海报，增加巨大裁切文字、规则线、纹理、侧边索引。
- `image_led`：图像主导海报，增加径向光、纹理线、裁切 SVG 视觉、侧边 meta、底部文字区。
- `event_info`：信息/活动海报，增加网格、左侧色条、斜向色块、信息章、数据行。
- `cta_campaign`：campaign 海报，增加分栏构图、背景网格、图形主视觉、强 CTA 条和微文案。

同时对插入 HTML 的用户文本做了 `html.escape`，降低 prompt 特殊字符破坏 HTML 的风险。

### 5. 增强无 VLM 时的视觉质量评审

修改文件：

- `backend/app/agents/vlm_critic.py`

优化点：

- VLM prompt 明确要求惩罚稀薄模板：普通渐变、居中文字、单个通用图标、空洞占位视觉、弱层级、无纹理/无层次。
- heuristic fallback 新增视觉层次检测：统计渐变、纹理、SVG、clip-path、mix-blend-mode、grid、规则线、frame、data-row 等视觉线索。
- 检测旧式“单个淡圆圈 SVG 占位图”并标为 `style` 问题，使路由进入下一轮优化。
- 该检查只在真实生成状态携带 `poster_brief` 或 `art_direction` 时启用，避免影响基础 HTML 单元测试。

### 6. 改善纯文字/极简 fallback 判断

修改文件：

- `backend/app/agents/content_extractor.py`

优化点：

- 识别真实中文和英文的纯文字/极简意图，如“纯文字”“只用文字”“极简”“no image”“type-only”等。
- 对纯文字海报将 `main_visual` 标记为 `omit`，避免所有海报都被塞进图像主导模板。
- 补充真实中文 CTA 识别与 CTA 文案生成，如“报名”“购买”“预约”“扫码”“注册”“关注”等。

## 2026-06-03 二次架构优化：视觉系统规划层 + 上下文/输出长度配置

### 问题判断

上一轮已经强化了提示词、fallback 模板和 critic，但“元素太少”的问题仍然可能出现，根因不是某个参数偏保守，而是链路里缺少一个可执行的中间产物：

- 当前链路从 `PosterBriefV2` + `ArtDirectionV2` 直接跳到 HTML/CSS，LayoutPlanner 只能根据抽象风格提示自由发挥，容易退化成“背景 + 居中标题 + 小图标”。
- “增加层次”之前只是 prompt 里的软要求，没有成为 `GraphState` 中可检查、可复用、可反馈修订的结构。
- VLM/heuristic critic 只能判断结果是否稀薄，但不能明确知道模型漏掉了哪些计划中的视觉层。
- 关于“加大上下文长度”：真正的模型上下文窗口由模型/供应商决定，代码不能凭空扩容；但项目可以调大输出 token 上限，并减少本地对 HTML/反馈上下文的主动截断。

### 优化方案

新增独立架构阶段：

```text
ContentExtractor
  -> StyleDirector
  -> VisualSystemPlanner   # 新增：规划可执行视觉层系统
  -> SpatialLayoutPlanner  # 按视觉系统计划生成 HTML/CSS
  -> HTMLPainter
  -> VLMCritic
  -> Router
```

核心思路：

- 把“海报要更丰富”升级为 `VisualSystemPlan`：明确构图类型、密度、目标层数、必需 HTML id、背景层、纹理层、主视觉、裁切字体、方向性形状、框线、metadata strip、CTA/QR 等。
- 每轮 layout 前都重新运行 `VisualSystemPlanner`，让上一轮 VLM 反馈先修订视觉蓝图，再生成 HTML。
- LayoutPlanner 必须消费 `VisualSystemPlan`，并实现 `required_html_ids`，避免把多个层合并成一个模糊背景。
- VLM prompt 和本地 heuristic critic 都读取 `VisualSystemPlan`，检查必需层是否落到 HTML 中。
- 对上下文/输出长度做配置化：调大 raw HTML 生成输出上限，并调大 VLM 审稿可见 HTML 片段和 layout 可见反馈片段。

## 2026-06-03 三次修复：定位最终烂图原因与兜底质量线

### 真实原因

这次用户看到的烂图不是前面设计 agent 没有规划，而是最终 HTML 生成阶段失败后静默落回了本地 fallback 模板。

证据：

- 最终 HTML 文件 `generated/5be61fe2bf334748addf1d2ead18ce11_0.html` 和 `_1.html` 都包含固定模板文案：
  - `CAMPAIGN`
  - `POSTAI`
  - `clear action, strong hierarchy, poster-first layout`
- 后端日志显示 `SpatialLayoutPlanner` 的 raw HTML LLM 调用在约 61 秒处失败：
  - `LLM FAIL | https://api.deepseek.com/chat/completions | duration=61059.8ms`
  - 第二轮同样在约 61 秒失败
- 因为 `ALLOW_MODEL_FALLBACK=true`，系统没有中断，而是调用 `_run_fallback()` 生成本地模板。
- VLM 实际上已经正确指出问题，评分 42/52，并明确说缺少薪资、福利、岗位要求、二维码、校园插画等。

结论：

```text
Content / Style / VisualSystem 都成功了
但 LayoutPlanner raw HTML 生成超时失败
于是最终图片来自 fallback 模板
所以日志里设计信息很多，结果却很差
```

### 关于视觉模型配置

检查 `.env` 后发现：

```ini
VISION_MODEL=qwen3-vl-flash
```

这不是普通语言模型。根据阿里云百炼官方文档，`qwen3-vl-flash` 属于 Qwen3-VL 视觉模型，并且兼容 OpenAI Vision 接口。

因此本次问题不在视觉模型名，而在 HTML 生成阶段超时 fallback。

### 已实现修复

#### 1. 增加 raw HTML 生成超时配置

修改文件：

- `backend/app/core/config.py`
- `backend/app/core/llm_client.py`
- `backend/.env`
- `backend/.env.example`

说明：

- 结构化 JSON agent 使用 `LLM_TIMEOUT_SECONDS`
- HTML/CSS raw 生成使用 `LLM_RAW_TIMEOUT_SECONDS`
- VLM 审稿使用 `VISION_TIMEOUT_SECONDS`
- 之前 raw HTML 默认约 60 秒，复杂海报很容易超时后 fallback

之前日志中 `error=` 可能为空。现在会打印异常类型，例如：

这样下次能直接看出是超时、连接失败还是 HTTP 错误。

#### 3. fallback 不再是空洞 campaign 模板

## 2026-06-03 四次架构优化：内容自问自答扩展层

### 问题判断

这次用户指出的问题是：即使视觉层、构图层有很多设计信息，最终音乐节海报仍然只有极少文字和元素。根因不只是视觉丰富度不足，而是内容链路过度依赖用户原始输入。

例如用户只输入：

```text
生成一张音乐节的海报
```

旧链路会倾向于只提取“音乐节”这个主题，再把缺失的日期、地点、阵容、票务、舞台、时间等信息直接省略。这样后续 StyleDirector / VisualSystemPlanner 即使设计得很热闹，也缺少可排版的信息骨架，最终容易变成“标题 + 少量装饰”的空海报。

因此这次优化不是继续调某个参数，而是在内容阶段新增一个独立的自问自答层：

```text
ContentExtractor
  -> ContentExpander       # 新增：根据海报类型补齐常见信息槽
  -> StyleDirector
  -> VisualSystemPlanner
  -> SpatialLayoutPlanner
  -> HTMLPainter
  -> VLMCritic
```

### 优化原则

- 保留用户明确给出的事实，不改写用户意图。
- 允许模型根据海报类型补“应该有的信息结构”，例如音乐节海报常见的日期、地点、阵容、舞台、票务。
- 不允许凭空编造精确事实。缺失事实只能用 `DATE TBD`、`VENUE TBD`、`LINEUP TBA`、`TICKETS INFO`、`SCHEDULE TBA` 等可编辑占位符。
- 对极简、纯文字、抽象-only、不要额外信息的 prompt 保持稀疏，不强行加内容。
- 扩展结果必须进入 `GraphState` 和最终 API response，方便解释“为什么加了这些字段”。

### 预期效果

- 用户输入很短时，系统不会只机械提取原文，而会根据海报类型补齐常见信息结构。
- 音乐节海报会天然具备可排版的信息层级：标题、日期、地点、阵容、舞台、票务、时间等。
- 缺失事实仍保持诚实，不会出现假日期、假地点、假艺人、假赞助商。
- 后续风格规划、视觉系统规划和 HTML 生成都会消费扩展后的 `poster_brief`，画面不再只靠装饰层撑丰富度。
- 即使 raw HTML 生成再次 fallback，本地模板也能保留扩展信息，不会退回只有标题和少量元素的空海报。
